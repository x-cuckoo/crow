# Challenge Proposals

Use this when the user already has a design or implementation idea.

- Do not treat the proposed solution as correct or optimal by default.
- Compare the proposed solution against the current code shape, constraints, and likely maintenance cost.
- If a better, safer, or simpler approach exists, propose it proactively.
- When rejecting or adjusting a proposal, explain the concrete tradeoff: correctness, scope, complexity, operability, or future change cost.
- If the proposal is good enough, say why it fits.
